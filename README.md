# LupiraCalWeb

Web frontend for [LupiraCalApi](https://github.com/danbro96/LupiraCalApi), consuming its REST surface only (never CalDAV). Its purpose is visualizing the temporal-backbone data no DAV client can see: calendar classes and kinds, typed item details, event-bound LLM prompt/action payloads, completeness scores, availability segments, inbox curation, tags, metadata, and cross-API relations.

## Shape

One Docker image holding two things:

- **`src/LupiraCalWeb`** — a .NET 10 BFF. Drives Authentik OIDC (Authorization Code + PKCE, `lupira-cal` client), keeps tokens server-side in an HttpOnly cookie session (`__Host-lupira-cal`), and proxies `/api/*` → LupiraCalApi via YARP with the user's bearer attached. The SPA never holds a token; there is no CORS.
- **`src/LupiraCalWeb.Client`** — a Vite + React 19 SPA served from the BFF's `wwwroot`. Online-only (React Query, the server is the source of truth). The API client and react-query hooks are **generated by orval** from LupiraCalApi's build-time OpenAPI doc (`backend-openapi.json`, refreshed via `npm run gen:api`).

SPA layering, enforced by `eslint-plugin-boundaries` (downward-only): `domain → data → state → ui` plus a leaf `config`.

## Screens

- **Calendar** (`/`) — hand-rolled month/week/day grids. Occurrences arrive recurrence-expanded from `GET /items?from&to`, one query per visible calendar (colors/filters per source calendar). Availability-kind calendars render as a background band; proposed items ghost dashed; System calendars toggle in the sidebar.
- **Item drawer** (`?item=<id>` on any route) — every REST-exposed field: times/RRULE, place (with map link), tags, typed kind details, ⚡ prompt/action payload editor (XOR, 409-aware), completeness ring + gaps, attendees with RSVP actions, calendar memberships + curation, relations, metadata merge editor.
- **Inbox** (`/inbox`) — the curation queue: items proposed into any calendar, accept/reject per membership.
- **Contacts** (`/contacts`) — search, reach fields (emails/phones/addresses/profiles), birthdays, groups & organizations, completeness.
- **Manage** (`/calendars`) — containers with class/kind/color/timezone/access, creation, and owner sharing by email.

## Develop

```bash
# API (sibling repo; Development accepts X-Dev-User)
cd ../LupiraCalApi && dotnet run --project src/LupiraCalApi     # http://localhost:5255

# BFF (dev auto-auth, forwards X-Dev-User, proxies /api → :5255)
dotnet run --project src/LupiraCalWeb                            # http://localhost:5181

# SPA (proxies /api + /auth to the BFF)
cd src/LupiraCalWeb.Client && npm install && npm run dev         # http://localhost:5174
```

| Script | Does |
| --- | --- |
| `npm run dev` / `build` / `preview` | Vite |
| `npm run lint` / `test` / `typecheck` | eslint / vitest / tsc |
| `npm run gen:api` | copy `../LupiraCalApi/docs/openapi/LupiraCalApi.json` → regenerate the orval client |

## Deploy

`docker build -t danbro96/lupira-cal-web .` (repo root). Runs at `https://cal.lupira.com` behind cloudflared; compose in `deploy/` (port 40881, API at 40880). Authentik: add redirect URI `https://cal.lupira.com/signin-oidc` to the `lupira-cal` client. Data-protection keys persist in the `cal-web-keys` volume so sessions survive restarts.
